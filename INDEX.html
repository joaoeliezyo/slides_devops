O que é Contêinerização?

Contêinerização é a tecnologia que permite empacotar uma aplicação e todas as suas dependências em um único artefato isolado. Conheça os conceitos e primitivas que tornam isso possível:

Isolamento (Namespaces)

Cria partições virtuais de recursos do sistema. O contêiner ganha sua própria árvore de processos (PID), interfaces de rede (NET), pontos de montagem (MNT) e usuários, rodando de forma isolada.

Kernel: namespaces

Controle (cgroups)

Control Groups limitam e medem o uso de recursos físicos. Define limites rígidos de consumo de CPU, memória RAM, I/O de disco e rede, protegendo o sistema contra sobrecargas.

Kernel: cgroups

Camadas (UnionFS)

O sistema de arquivos é composto por camadas imutáveis sobrepostas. Uma camada temporária de escrita (Read-Write) é adicionada no topo ao iniciar, usando Copy-on-Write (CoW).

Filesystem: UnionFS

Kernel Compartilhado

Diferente de Máquinas Virtuais (VMs), os contêineres compartilham diretamente o mesmo Kernel do sistema operacional hospedeiro. Isso elimina o overhead de emular hardware virtual.

Architecture: Shared Kernel

Empacotamento (Imagens)

Consolida o código, runtime (Node, Python), dependências, configurações e ferramentas do sistema em um único pacote imutável, garantindo consistência absoluta entre ambientes.

Artifact: OCI Image

Portabilidade Total

Como o contêiner traz consigo todo o necessário para rodar, ele funciona de forma consistente em qualquer infraestrutura: do seu laptop local a servidores bare-metal ou nuvem.

Standard: Run Anywhere

guest@docker:~$ ./compare_architectures.sh --vm --container

VM vs. Container

Ambos isolam aplicações, mas de formas radicalmente diferentes. VMs virtualizam hardware inteiro. Containers compartilham o kernel do SO — muito mais eficientes.

Característica VM Container
Abstração Hardware físico Kernel do SO
Tamanho típico 1 – 20 GB 5 – 500 MB
Tempo de boot 30 s – 5 min < 1 segundo
SO próprio Sim (completo) Não (compartilha kernel)
Overhead de CPU Alto (emulação) Quase zero
Overhead de memória Alto (SO convidado) Mínimo
Isolamento Forte (hardware) Bom (namespaces)
Portabilidade Limitada Total (qualquer host)

O que é o Docker?

Docker é a plataforma open-source líder mundial que revolucionou a engenharia de software ao padronizar como criamos, empacotamos e executamos aplicações em qualquer infraestrutura através de contêineres.

Build (Construir)

Escreva receitas sequenciais em arquivos de texto Dockerfile para compilar e empacotar seu código de forma determinística em imagens portáveis.

Standard: Dockerfile

Share (Compartilhar)

Publique e distribua suas imagens no Docker Hub ou em Registries privados corporativos, facilitando a automação de pipelines de CI/CD.

Platform: Docker Hub

Run (Executar)

Inicie instâncias seguras e totalmente isoladas (contêineres) a partir de imagens em segundos, aproveitando a performance nativa do Kernel host.

Runtime: dockerd / containerd

Conceitos Básicos

Três conceitos fundamentais formam a base de todo workflow Docker. Clique nas abas para explorar cada um em profundidade.

O Dockerfile é a receita da sua imagem — um arquivo de texto com instruções sequenciais. Cada instrução cria uma nova camada na imagem. Clique em uma linha para ver a explicação.

1234 5678 9101112
# Dockerfile — Aplicação Node.js
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]
Ordem importa
Instruções menos mutáveis no topo preservam cache de build
Cache por camada
Se uma instrução não mudou, Docker reutiliza a camada em cache
Multi-stage builds
Use FROM múltiplos para imagens de prod menores e mais seguras

Enquanto o Dockerfile é a receita de construção, a Imagem é o pacote imutável compilado. Ela materializa na prática o conceito de camadas sobrepostas do UnionFS apresentadas no início. Cada instrução do Dockerfile gera uma camada de leitura estática (Read-Only) identificada por um hash único. A camada temporária gravável (Read-Write) só é criada quando o container é de fato executado.

Camada R/W do Container READ-WRITE
CMD node server.js +1 KB
RUN npm ci +48 MB
COPY . . +2 MB
WORKDIR /app +1 KB
FROM node:20-alpine +7 MB
Copy-on-Write (CoW): Quando um container tenta modificar um arquivo das camadas de leitura da imagem, o Docker copia apenas esse arquivo para a camada R/W antes de editá-lo. As camadas originais da imagem nunca são alteradas — por isso múltiplos containers podem compartilhar a mesma imagem sem conflito.
Imutável
Camadas da imagem nunca mudam após o build
Compartilhada
100 containers podem usar 1 imagem. Zero duplicação.
Versionada por tag
meu-app:v1.0, meu-app:latest, meu-app:sha-a3f2b1

Um container é a instância ativa e isolada de uma imagem. Sob o capô, ele é apenas um processo comum rodando diretamente no kernel host, mas blindado via Namespaces e com limites rígidos definidos por cgroups (as primitivas explicadas no primeiro slide). Você pode instanciar dezenas de containers a partir da mesma imagem, pois cada um possui sua própria camada gravável isolada.

Created
docker create
Container criado, ainda não iniciado. Camada R/W alocada.
Running
docker start / run
Processo em execução. Consome CPU e memória do host.
Paused
docker pause
Processos congelados via SIGSTOP. Memória preservada.
Stopped
docker stop
SIGTERM → grace period → SIGKILL. Container parado mas existe.
Removed
docker rm
Container deletado. Camada R/W destruída. Use volumes para persistir dados.
terminal — comandos essenciais
# Criar e iniciar container (tudo em um) $ docker run -d -p 3000:3000 --name minha-api meu-app # Listar containers rodando $ docker ps # Ver logs em tempo real $ docker logs -f minha-api # Entrar no container (shell interativo) $ docker exec -it minha-api sh # Inspecionar uso de recursos $ docker stats minha-api # Parar e remover $ docker stop minha-api && docker rm minha-api

Linha do Tempo do Docker

Conheça a história e os marcos evolutivos que levaram um projeto interno de uma startup francesa em 2013 a se tornar o alicerce absoluto de toda a nuvem moderna.

Março de 2013
Solomon Hykes apresenta o Docker como projeto open-source na conferência PyCon. Inicialmente um componente interno da PaaS francesa dotCloud.
Junho de 2014
Lançamento oficial do Docker 1.0. A dotCloud muda oficialmente o seu nome corporativo para Docker Inc. e a adoção pela comunidade de desenvolvimento de software explode globalmente.
Junho de 2015
Criação da Open Container Initiative (OCI) pelas principais líderes do mercado (Docker, Google, Red Hat, AWS, Microsoft), com foco em padronizar as especificações de imagem e execução (runtime) do setor.
Hoje
Mais de 14 bilhões de downloads mensais de imagens realizados no Docker Hub. Tornou-se o pilar padrão do movimento DevOps e da orquestração de microsserviços modernos.

O Ecossistema Docker

Entenda como os principais blocos do ecossistema do Docker se articulam para buildar, armazenar e rodar suas aplicações de forma orquestrada.

Docker CLI
Interface de linha de comando. O que você digita no terminal. Envia comandos para o Daemon via socket Unix.
# O ponto de entrada para orquestrar: $ docker system info $ docker network ls $ docker volume ls $ docker compose up -d
Docker Daemon (dockerd)
Processo em background que faz o trabalho pesado: gerencia imagens, containers, redes e volumes. Usa containerd + runc internamente.
# Socket de comunicação /var/run/docker.sock # Componentes internos dockerd → containerd → runc
Registry (Docker Hub)
Repositório de imagens. Pense no npm para pacotes, mas para imagens Docker. Pode ser público ou privado.
# Baixar imagem do Hub $ docker pull nginx:alpine # Enviar sua imagem $ docker push user/meu-app:v1
Redes
Comunicação entre containers via bridge, host ou overlay networks.
# Criar uma rede bridge customizada $ docker network create app-net # Iniciar container conectado à rede $ docker run -d --net app-net --name web nginx # Listar todas as redes no host $ docker network ls
Volumes
Persistência de dados além do ciclo de vida do container.
# Criar volume nomeado persistente $ docker volume create pg-data # Montar volume no container (postgres) $ docker run -d -v pg-data:/var/lib/postgresql/data postgres # Inspecionar detalhes do volume $ docker volume inspect pg-data
Docker Compose
Orquestra múltiplos containers com um único arquivo YAML.
# Subir stack completa em background $ docker compose up -d # Visualizar status dos serviços compose $ docker compose ps # Derrubar stack e remover volumes $ docker compose down -v

Vantagens no Desenvolvimento

Docker não é só uma ferramenta de deploy — transforma todo o ciclo de desenvolvimento, do setup local ao CI/CD em produção.

Ambiente 100% Reproduzível
Todo desenvolvedor do time roda exatamente o mesmo ambiente. Sem "funciona na minha máquina". O Dockerfile é a fonte da verdade.
# Novo dev no time? Só isso: $ git clone repo && docker compose up # Ambiente idêntico em 2 minutos ✓
Isolamento de Dependências
Projetos diferentes podem usar versões conflitantes da mesma tecnologia no mesmo servidor, sem interferência.
# Projeto A — Node 16 docker run node:16 node -v # Projeto B — Node 20 (mesmo host!) docker run node:20 node -v
CI/CD Simplificado
Build uma vez, execute em qualquer lugar. A imagem gerada no pipeline é exatamente a mesma que vai para produção.
# Pipeline GitHub Actions - name: Build e Push run: | docker build -t app:$SHA . docker push app:$SHA # Mesma imagem → staging → prod
Escalabilidade Horizontal
Inicie dezenas de instâncias idênticas em segundos para lidar com picos de tráfego. Cada container é stateless e substituível.
# Sobe 5 instâncias do serviço web $ docker compose up --scale web=5 # Resultado: web_1, web_2, web_3, web_4, web_5 ✓
Isolamento de Segurança
Cada container opera em namespaces isolados. Uma vulnerabilidade em um serviço não compromete automaticamente os outros.
# Limitar recursos do container docker run \ --memory="512m" \ --cpus="0.5" \ --read-only \ meu-app
Infraestrutura como Código
Dockerfile e docker-compose.yml vivem no repositório junto com o código. Infraestrutura versionada, revisada e auditável.
# Tudo no git git log --oneline a3f2b1c fix: atualiza Node 20→22 7d9e4f0 feat: adiciona serviço Redis 2c8a1b3 chore: reduz imagem (multi-stage)

guest@docker:~$ cat docker-compose.yml
